CONCIDERACIONES PARA TABLA DE REFERIDOS 
Para tu sistema MLM de 3 niveles en Supabase (PostgreSQL), la forma más fácil, efectiva y con mejor rendimiento es usar Consultas Recursivas (CTE - Common Table Expressions) . Esta es la herramienta nativa de PostgreSQL para manejar estructuras jerárbicas como árboles de referidos.

Aquí tienes las soluciones prácticas para ambos casos (ver hacia abajo y hacia arriba), listas para implementar.

---

📊 Solución 1: Ver la red HACIA ABAJO (Mis referidos en niveles 1,2,3)

Esta consulta te devuelve todos los usuarios que están debajo de un referidor específico, limitado a 3 niveles de profundidad.

```sql
-- Función para obtener los referidos hacia abajo de un usuario
CREATE OR REPLACE FUNCTION get_downline_referrals(p_user_id UUID, max_level INT DEFAULT 3)
RETURNS TABLE (
    referral_id UUID,
    username TEXT,
    level INT,
    path TEXT[]
) LANGUAGE plpgsql AS $$
BEGIN
    RETURN QUERY
    WITH RECURSIVE referral_tree AS (
        -- Nivel 1: Referidos directos
        SELECT 
            id,
            username,
            1 as level,
            ARRAY[id] as path
        FROM profiles
        WHERE sponsor_id = p_user_id

        UNION ALL

        -- Niveles siguientes (recursivo)
        SELECT 
            p.id,
            p.username,
            rt.level + 1,
            rt.path || p.id
        FROM profiles p
        INNER JOIN referral_tree rt ON p.sponsor_id = rt.referral_id
        WHERE rt.level < max_level  -- Controla profundidad máxima
    )
    SELECT 
        referral_id,
        username,
        level,
        path
    FROM referral_tree
    ORDER BY level, username;
END;
$$;

-- Uso: obtener referidos del usuario con ID 'xxx' (3 niveles)
SELECT * FROM get_downline_referrals('usuario-id-aqui', 3);
```

¿Qué devuelve? 

· referral_id: ID del referido
· username: nombre del usuario
· level: nivel exacto (1,2,3)
· path: ruta completa (ej. {A,B,C} para mostrar jerarquía)

⬆️ Solución 2: Ver la red HACIA ARRIBA (Sponsors para comisiones)

Esta consulta te devuelve los patrocinadores (upline) de un usuario hasta 3 niveles arriba, ideal para calcular comisiones cuando alguien deposita o gana.

```sql
-- Función para obtener los sponsors hacia arriba de un usuario
CREATE OR REPLACE FUNCTION get_upline_sponsors(p_user_id UUID, max_level INT DEFAULT 3)
RETURNS TABLE (
    sponsor_id UUID,
    username TEXT,
    level INT  -- 1 = sponsor directo, 2 = sponsor del sponsor, etc.
) LANGUAGE plpgsql AS $$
BEGIN
    RETURN QUERY
    WITH RECURSIVE upline_tree AS (
        -- Caso base: buscar sponsor directo
        SELECT 
            id as sponsor_id,
            username,
            1 as level
        FROM profiles
        WHERE id = (SELECT sponsor_id FROM profiles WHERE id = p_user_id)
        AND sponsor_id IS NOT NULL

        UNION ALL

        -- Niveles superiores
        SELECT 
            p.id,
            p.username,
            ut.level + 1
        FROM profiles p
        INNER JOIN upline_tree ut ON p.id = (SELECT sponsor_id FROM profiles WHERE id = ut.sponsor_id)
        WHERE ut.level < max_level
    )
    SELECT 
        sponsor_id,
        username,
        level
    FROM upline_tree
    ORDER BY level;
END;
$$;

-- Uso: obtener sponsors del usuario 'xxx' (3 niveles arriba)
SELECT * FROM get_upline_sponsors('usuario-id-aqui', 3);
```

¿Qué devuelve?

· sponsor_id: ID del patrocinador
· username: nombre
· level: 1=directo, 2=indirecto, 3=tercer nivel

📁 Tabla referrals (o referral_links)

Esta tabla almacena explícitamente cada relación patrocinador-referido, lo que permite consultar la red sin recursión en tiempo real.

Campos recomendados:

Campo Tipo Descripción
id uuid (PK) Identificador único de la relación.
sponsor_id uuid (FK → profiles.id) El usuario que patrocina.
referred_id uuid (FK → profiles.id) El usuario referido.
level smallint Nivel de profundidad (1, 2, 3…). Opcional, pero acelera consultas de nivel fijo.
created_at timestamptz Fecha en que se creó la relación.

Índices clave:

```sql
-- Para búsquedas rápidas descendentes (patrocinador → sus referidos)
CREATE INDEX idx_referrals_sponsor ON referrals(sponsor_id);

-- Para búsquedas ascendentes (referido → su patrocinador directo)
CREATE INDEX idx_referrals_referred ON referrals(referred_id);

-- Índice compuesto si necesitas filtrar por nivel
CREATE INDEX idx_referrals_sponsor_level ON referrals(sponsor_id, level);

-- Garantizar que no haya duplicados (un usuario solo puede ser referido una vez)
CREATE UNIQUE INDEX idx_referrals_unique_referred ON referrals(referred_id);
```

Mantenimiento automático con triggers:

Para no depender de que la aplicación inserte correctamente en esta tabla, puedes crear un trigger que, cada vez que se inserte un nuevo perfil con sponsor_id, genere automáticamente todas las relaciones hacia arriba (niveles 1, 2, 3).

```sql
-- Función que se ejecuta al insertar un perfil
CREATE OR REPLACE FUNCTION maintain_referral_tree()
RETURNS TRIGGER AS $$
DECLARE
    current_sponsor_id uuid;
    current_level int;
BEGIN
    -- Insertar la relación directa (nivel 1)
    IF NEW.sponsor_id IS NOT NULL THEN
        INSERT INTO referrals (sponsor_id, referred_id, level)
        VALUES (NEW.sponsor_id, NEW.id, 1);

        -- Ahora subimos al sponsor del sponsor (nivel 2)
        current_sponsor_id := NEW.sponsor_id;
        current_level := 2;

        WHILE current_level <= 3 AND current_sponsor_id IS NOT NULL LOOP
            -- Buscar el sponsor del sponsor actual
            SELECT sponsor_id INTO current_sponsor_id
            FROM profiles
            WHERE id = current_sponsor_id;

            IF current_sponsor_id IS NOT NULL THEN
                INSERT INTO referrals (sponsor_id, referred_id, level)
                VALUES (current_sponsor_id, NEW.id, current_level);
            END IF;

            current_level := current_level + 1;
        END LOOP;
    END IF;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Trigger después de insertar en profiles
CREATE TRIGGER trg_after_insert_profile
AFTER INSERT ON profiles
FOR EACH ROW
EXECUTE FUNCTION maintain_referral_tree();
```

Con esta tabla, consultar los referidos de nivel 1, 2 o 3 de un usuario se vuelve instantáneo:

```sql
-- Obtener todos los referidos de nivel 1 de un usuario
SELECT referred_id FROM referrals WHERE sponsor_id = 'usuario-id' AND level = 1;

-- Obtener todos los referidos de nivel 2 y 3 (para comisiones residuales)
SELECT referred_id FROM referrals WHERE sponsor_id = 'usuario-id' AND level IN (2,3);
```

⚡ Optimizaciones generales para todas las tablas

1. Índices estratégicos

Crea índices en:

· Todas las foreign keys (ej. user_id en wallets, deposits, etc.).
· Columnas usadas en búsquedas frecuentes (status, date, level).
· Columnas usadas en ordenamientos (created_at descendente).

Ejemplos:

```sql
-- profiles
CREATE INDEX idx_profiles_sponsor ON profiles(sponsor_id);
CREATE INDEX idx_profiles_referral_code ON profiles(referral_code);

-- deposits
CREATE INDEX idx_deposits_user_status ON deposits(user_id, status);
CREATE INDEX idx_deposits_created_at ON deposits(created_at DESC);

-- daily_earnings
CREATE INDEX idx_daily_earnings_user_date ON daily_earnings(user_id, date);
```

2. Tipos de datos adecuados

· Usa numeric para dinero (evita float por errores de redondeo).
· Usa timestamptz para fechas (maneja zonas horarias automáticamente).
· Para textos cortos y fijos, varchar(n); para largos, text.
· Las claves primarias: uuid es seguro y distribuible; si la tabla es enorme, considera bigserial (entero autoincremental) para ahorrar espacio.

3. Evita NULL en lo posible

· Los campos que puedan ser nulos (como sponsor_id en el primer usuario) deben permitirlo, pero en búsquedas, los índices no incluyen NULLs a menos que uses índices parciales. Si necesitas buscar NULL, crea un índice parcial:
    CREATE INDEX idx_profiles_sponsor_null ON profiles(id) WHERE sponsor_id IS NULL;

4. Normalización vs Desnormalización

· La tabla referrals es una desnormalización controlada que mejora el rendimiento de lecturas a costa de mantenerla con triggers.
· Puedes aplicar desnormalización similar en otras áreas, por ejemplo, almacenar el saldo total en profiles (en lugar de calcularlo siempre desde wallets), pero actualízalo con triggers.

5. Particionamiento para tablas grandes

· Tablas como transactions, daily_earnings o deposits pueden crecer rápidamente. En Supabase (PostgreSQL 12+) puedes particionar por rango de fechas.
· Ejemplo para daily_earnings:
  ```sql
  CREATE TABLE daily_earnings (
      id uuid,
      user_id uuid,
      date date,
      amount numeric,
      PRIMARY KEY (id, date)
  ) PARTITION BY RANGE (date);
  ```
  Luego creas particiones mensuales. Las consultas con filtro por fecha accederán solo a la partición relevante.

6. Uso de vistas materializadas

· Si hay reportes que se consultan con frecuencia pero no necesitan datos en tiempo real (ej. ranking de ganadores), crea vistas materializadas y actualízalas con un cron job diario.
· Ejemplo: vista de referidos con nivel precalculado (similar a la tabla referrals pero quizás más completa).

7. Configuración de RLS (Row Level Security)

· En Supabase, cada tabla debe tener políticas RLS. Asegúrate de que las políticas usen índices. Por ejemplo, una política como USING (user_id = auth.uid()) se beneficia del índice en user_id.

8. Mantenimiento periódico

· Ejecuta VACUUM y ANALYZE regularmente (Supabase lo hace automáticamente, pero en tablas muy activas puede ayudar hacerlo manual en horas de baja carga).
· Monitorea consultas lentas con pg_stat_statements.

9. Evita funciones pesadas en consultas frecuentes

· Si usas funciones recursivas (CTE) para calcular el árbol, hazlo solo cuando sea necesario. Prefiere la tabla referrals para consultas frecuentes.
· Para cálculos de comisiones, puedes crear funciones PL/pgSQL que usen la tabla referrals y operen en lotes, no fila por fila.

---

🚀 Recomendación final

1. Empieza con la tabla referrals y los triggers desde el principio. Esto hará que las consultas de red sean ultrarrápidas y escalables.
2. Aplica índices en todas las tablas según las columnas de búsqueda y relación.
3. Monitorea el rendimiento con las herramientas de Supabase (GraphQL, logs) y ajusta índices si ves consultas lentas.
4. Considera el particionamiento cuando las tablas superen el millón de registros.
